# RAG-пайплайн и обработка новости

## Полный конвейер

```
Парсинг → Сохранение сырых данных → Дедупликация → Эмбеддинг →
RAG (поиск похожих + промпт) → LLM → Сохранение результата → Отправка
```

### Шаги подробно

1. **Парсинг** — `scheduler.py` запускает задачи по расписанию для каждого источника каждого клиента.
2. **Сохранение сырых данных** — запись в `data/clients/<id>/raw/YYYY-MM-DD.jsonl` и в таблицу `News` (с хэшем контента).
3. **Дедупликация** — проверка по хэшу (`News.hash`), затем TF-IDF + cosine similarity для семантических дубликатов.
4. **Генерация эмбеддинга** — `processors/embeddings.py`, модель `paraphrase-multilingual-MiniLM-L12-v2`. Выполняется через `asyncio.to_thread`.
5. **Сохранение в ChromaDB** — коллекция именуется `client_{client_id}`.
6. **RAG-поиск** — 3–5 похожих новостей из коллекции клиента для few-shot контекста.
7. **Формирование промпта** и отправка в Ollama (Saiga Llama 3 8B).
8. **Сохранение результата** — `summary`, `sentiment`, `entities`, `hashtags` → таблица `News`.
9. **Отправка пользователю** — если новость соответствует ключевым словам и не дубликат. С inline-кнопками (интересно / неинтересно / сохранить).
10. **Обработка фидбека** — реакции сохраняются в `Feedback`, влияют на будущую фильтрацию.

## Шаблон промпта

```
Ниже приведены несколько похожих новостей и их краткое содержание (примеры).

Пример 1: {текст похожей новости}
Краткое содержание: {саммари}

Пример 2: {текст похожей новости}
Краткое содержание: {саммари}

Теперь сделай краткое содержание (2-3 предложения), определи тональность
(позитив/негатив/нейтрально) и предложи 3-5 хештегов для следующей новости:

НОВОСТЬ: {текст}

Краткое содержание:
Тональность:
Хештеги:
```

## Дедупликация

1. **Быстрая:** сравнение хэша (`News.hash`) — O(1), до сохранения в БД.
2. **Семантическая:** TF-IDF + cosine similarity через `sklearn` — для похожих, но не идентичных текстов.

Порог схожести (например, 0.85) задаётся в конфиге клиента (`analysis.deduplication_threshold`).
